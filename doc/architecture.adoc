== Four-Layer Model

Rift/1 splits the real‑time stack into four layers (spec §3):

[horizontal]
Transport::  Carries bidirectional byte‑streams or datagrams (WebSocket, WebTransport, TCP).
Session::    Identity, connection lifetime, recovery, heartbeat, flow‑control negotiation.
Message::    Message type, reliability, ordering, ack, replay, priority, TTL.
Topic::       Subscription, publish, permission, sharding, fanout, retention, snapshot.

== Workspace Crate Map

[cols="2m,1m,6a",options="header"]
|===
| Crate | Layer | Responsibility

| `rifts-core`
| Message / Session / Topic
| Protocol fundamentals: frames, codecs, message models, protocol constants, topic profiles, metrics, errors, ack, and flow-control primitives. This crate has no transport, broker, or storage backend implementation.

| `rifts-storage`
| Topic
| Storage traits and local implementations: `StorageEngine`, memory stores, sled stores, key encoding, log, offset, dedupe, and snapshot APIs. Redis adapters intentionally live outside this crate.

| `rifts-transport`
| Transport
| Transport traits plus optional WebSocket/framework adapters for tungstenite, axum, actix-web, warp, and ntex.

| `rifts-session`
| Session
| Authentication, session identity, session state, offset tracking, resume decisions, and session store.

| `rifts-broker`
| Topic / Session
| Message routing core: `Broker`, `InMemoryBroker`, fanout, topic routing, and per-connection state machine.

| `rifts-redis`
| Topic / Infra
| Redis-backed storage adapters, Redis-backed broker, and Redis Pub/Sub fanout bridge for multi-instance deployments.

| `rifts-client`
| Client
| Async client SDK: WebSocket connection, handshake, reconnect, heartbeat, publish/subscribe helpers, typed client events.

| `rifts`
| —
| Umbrella facade crate. Re-exports the public API surface and provides `RiftServer` / `RiftServerBuilder`.
|===

The crate dependency direction is intentionally one-way:

[source]
----
rifts-storage   -> rifts-core
rifts-transport -> rifts-core
rifts-session   -> rifts-core, rifts-storage
rifts-broker    -> rifts-core, rifts-storage, rifts-session, rifts-transport
rifts-redis     -> rifts-core, rifts-storage, rifts-broker
rifts-client    -> rifts-core, rifts-transport
rifts           -> facade over the ecosystem crates

rifts-client depends on rifts-core and the WebSocket transport only.
rifts re-exports the ecosystem crates behind feature flags.
----

== Broker Architecture

The `Broker` trait is the central abstraction — all topic operations go through it:

[source,rust]
----
#[async_trait]
pub trait Broker: Send + Sync {
    async fn publish(&self, frame: &Frame) -> Result<PublishOutcome>;
    async fn subscribe(&self, topic: &str, intent: SubscribeIntent, sink: ConnectionSink)
        -> Result<SubscriptionId>;
    async fn unsubscribe(&self, id: SubscriptionId) -> Result<bool>;
    async fn drop_sink(&self, sink_id: u64) -> usize;
    async fn replay(&self, topic: &str, from: i64, to: i64) -> Result<Vec<Bytes>>;
    async fn snapshot(&self, topic: &str) -> Result<Option<StoredSnapshot>>;
    async fn subscriber_count(&self, topic: &str) -> usize;
    async fn head_offset(&self, topic: &str) -> i64;
}
----

The broker crate ships the in-process implementation. Redis-backed multi-instance
routing lives in `rifts-redis`.

[cols="2,3,5a",options="header"]
|===
| Impl | Use Case | Key Property

| `InMemoryBroker<O,L,D,S>`
| Single‑process, default
| Generic over storage backends.  `DefaultBroker` = all memory.  `SledBroker` = all sled.
|===

== Flow Control

Two cooperating mechanisms guard the server against overload:

=== BackpressureController

`BackpressureController` (spec §18.1) guards the per‑connection send queue with an
atomic CAS‑loop enqueue.  When the queue exceeds the high‑water mark (90% of
`max_send_queue_bytes`), the controller selects a strategy:

[cols="2,5a",options="header"]
|===
| Strategy | Behaviour

| `Pause` (default)
| Caller spins with `tokio::time::sleep` until capacity frees.

| `DropVolatile`
| Messages with `Priority::Volatile` or `Priority::Background` are discarded.

| `CoalesceState`
| State messages for the same `state_key` are collapsed to the latest.

| `Downgrade`
| All messages are deferred; the writer is throttled.

| `Disconnect`
| The slow consumer is kicked with `CloseCode::SlowConsumer`.

| `SnapshotLater`
| The connection switches to periodic snapshot polling instead of live messages.
|===

Each controller tracks local counters (`applied`, `dropped`, `slow_consumer`,
`flow_pause`) which feed into the process-wide `Metrics`.

=== RateLimiter

`RateLimiter` (spec §18.2) is a standard token‑bucket implementation used at two
levels:

- *Per‑connection*: limits the inbound publish rate from a single client.
- *Per‑topic*: limits the aggregate publish rate to a single topic (configured
  via `TopicProfile::rate_limit`).

[source,rust]
----
let limiter = RateLimiter::new(1_000, 2_000); // 1 000 msg/s, burst 2 000
if limiter.try_take() {
    // proceed
} else {
    // reject or back-off
}
----

== Metrics (spec §23.2)

`Metrics` is a process‑wide set of `AtomicU64` counters.  In production you would
export them to Prometheus or OTLP; the structure mirrors the spec so the mapping is
straightforward.

[cols="2,5a",options="header"]
|===
| Category | Counters

| Connections
| `active_connections`, `connection_open_total`, `connection_close_total`,
  `reconnect_total`, `resume_success_total`, `resume_failed_total`,
  `heartbeat_timeout_total`

| Messages
| `messages_in_total`, `messages_out_total`, `messages_dropped_total`,
  `messages_replayed_total`, `messages_expired_total`, `ack_timeout_total`,
  `duplicate_total`

| Backpressure
| `send_queue_depth`, `recv_queue_depth`, `slow_consumer_total`,
  `flow_pause_total`, `flow_resume_total`, `volatile_drop_total`,
  `state_coalesce_total`
|===

All counters use `Ordering::Relaxed` — they are approximate by design and intended
for monitoring dashboards, not control flow.

== Trace Identifier (spec §23.1)

The current implementation stores distributed trace correlation as the optional
`Frame::trace_id` field. It does not expose a dedicated trace context module.

When the `TRACE` frame flag is set, the payload may include richer trace metadata
defined by the protocol, but the typed Rust API currently models only the trace
chain identifier.

== Persistence Layer

[cols="2,5a",options="header"]
|===
| Trait | Key Methods | Memory Impl | Sled Impl

| `StorageEngine`
| `get`, `put`, `delete`, `scan_prefix`
| `MemoryEngine` (DashMap)
| `SledEngine` (sled::Tree)

| `OffsetStore`
| `alloc(topic) -> i64`, `head(topic) -> i64`
| `MemoryOffsetStore`
| `SledOffsetStore`

| `LogStore`
| `append(topic, entry, retention)`, `range(topic, from, to) -> Vec<LogEntry>`
| `MemoryLogStore`
| `SledLogStore`

| `DedupeStore`
| `check_and_record(topic, key, window) -> bool`, `sweep() -> usize`
| `MemoryDedupeStore`
| `SledDedupeStore`

| `SnapshotStore`
| `capture(topic, store, ttl)`, `get(topic)`, `list()`
| `MemorySnapshotStore`
| `SledSnapshotStore`
|===

Each sled store uses a separate `sled::Tree` within the same `Db` file, keyed with
`\x00`‑separated topic prefixes for efficient `scan_prefix` range scans.

== Error Model

Errors use `thiserror` deeply.  The top‑level `RiftError` composes six typed sub‑enums:

[source,rust]
----
pub enum RiftError {
    Frame(FrameReject),       // version, codec, schema, encoding
    Session(SessionReject),   // not found, expired, conflict, resume
    Topic(TopicReject),       // not found, closed, rate-limited, forbidden
    Message(MessageReject),   // duplicate, expired, too large, ack timeout
    Auth(AuthReject),         // required, invalid, expired, revoked, denied
    System(SystemReject),     // overloaded, maintenance, shard-moved, internal
    Config(ConfigError),      // invalid configuration
    Io(std::io::Error),
}
----

Large error sources (tungstenite, ciborium, serde_json) are boxed inside a `BoxedStdError`
wrapper so the `Result` type stays small on the stack.

== Testing

The workspace ships three test layers:

. *Unit tests* — inline `#[cfg(test)] mod tests` across the protocol, storage,
  broker, session, transport, Redis, and client crates.
. *Integration tests* — `rifts/tests/integration.rs` and
  `rifts/tests/redis_integration.rs`.
. *Doc tests* — public API examples in the library crates.

Run them all with:

[source,sh]
----
cargo test
cargo test -p rifts --features sled
cargo test --workspace --all-features
----
